## Session 2 — 2026-06-04 (Update — Homepage / Location / Site refinements)

Intent: Update. Scope: 10 issues across Homepage, Location detail, Site detail.
Sources for this session: live codebase audit (4 parallel Explore agents) + user direction.

### Codebase findings grounding this session
- F1 — `animalTags` is computed at build time from 6 hardcoded regex buckets in `src/lib/atlas-location.ts:113-123`. Underlying data is far richer: 536 distinct species across 356 sites. Filter can support ~12–14 categorised tags.
- F2 — Locations carry NO `heroImageUrl` (0/113). `atlas-location.ts:105-110` already borrows a quality underwater photo from the location's own dive sites (`isUnderwaterQualityPhoto` gate; sites have 100% coverage). The location page + inspiration grid never RENDER the borrowed photo — they draw CSS gradients only. Fix is wiring, not sourcing; borrowed photo is from the location's sites so it matches the place.
- F3 — `where-to-see/[species]` renders real underwater photos from `encounters.json heroImageUrl` (Wikimedia, 10/11). This is the "good photo" reference standard.
- F4 — Location planning sidebar (`locations/[slug]/page.tsx:1473-1663`): Operators block is visually dominant (dark `#0b1e32`, white, large heading, CTA button) > Getting there (medium) > Lodging (lowest). A hardcoded "See trip options" button always links to generic PADI search `padi.com/dive-shop-search?q={name}` — this is the dead Ogasawara link. `isAffiliate` flag exists in data but is not used to gate display.
- F5 — Lodging data carries `kind: "hotel" | "liveaboard"` and `priceLevel`. Liveaboards inherently include diving.
- F6 — Site detail sidebar (`sites/[slug]/page.tsx:1094-1484`) order: Depth profile → Part of [location] → Thermal status (conditions) → Operators. Conditions before booking = the confusing flow. No unified meta-badges row; a horizontal stat strip above both columns serves that role.
- F7 — Gear section fully removed from site UI (redesign commit `73258fb`), but data layer intact: `Site.gearIds`, `Site.siteSpecificGear` (`SiteGearItem[]`). Prior `SiteGearSection` rendered basic kit + site-specific add-ons as Amazon affiliate links.

### D06 — [Homepage] Location-card photo overlays (Issue 1) [CONFIRMED]

### D07 — [Homepage] Expand wildlife filter to categorised taxonomy (Issue 2)
Source: user ("as many as possible, with drop downs to categorise"; "check against what we have… is that enough or can there be more").
Decision: Expand from 6 flat tags to a categorised taxonomy grounded in actual site species (F1). Tags must be backed by real `animalTags` derivation — expand the regex buckets in `atlas-location.ts` so each new tag maps to species actually present. Proposed groups (final list pending data-coverage check at Finalize): Sharks (+ Hammerheads), Rays & mantas, Whales, Dolphins, Sea turtles, Seals & sea lions, Large pelagics/game fish, Reef fish, Cephalopods, Macro & critters, Corals & invertebrates. Each tag must resolve to ≥1 location or it is dropped (no empty filters). Wildlife becomes a grouped/collapsible facet, not a flat list.

### D10 — [Homepage + Location] Render real underwater photos (Issues 5 & 6)
Source: user ("whereas [where-to-see] has good photo. i want it to match the location where it's from. and underwater photo only").
Decision: Inspiration-grid cards and the location-detail hero must render the borrowed underwater site photo (F2) instead of CSS gradients. Photo source = the location's own dive sites, gated by `isUnderwaterQualityPhoto` (underwater-only, per memory `hero_must_be_underwater`). ### D11 — [Location] Restructure the planning flow (Issues 7 & 8)
Source: user ("order and different types of info"; operators over-highlighted vs accommodation; "if accommodation can book dives, operator's not needed"; "i want to know how to get there, what i need to book — accommodation, operator or both").
Decision: Rebuild the planning sidebar as a single "Plan your trip" block with logical order and EQUAL visual weight across booking types:
  1. Getting there (how to get there) — leads.
  2. What to book — accommodation and operators presented as peers, not a dominant dark operator block over a quiet lodging list.
Liveaboard nuance (F5): when a liveaboard lodging covers diving, surface it as a combined "stay + dive" option and suppress a redundant separate operator. Drop the hardcoded generic-PADI "See trip options" button (the dead Ogasawara link). Operators shown only when backed by a real/affiliate URL (F4 `isAffiliate` or a non-template URL) — never a synthesized generic search. If no real operator and no dive-capable lodging, show getting-there + accommodation only; do not invent operators. Witnessing-change muted-CTA treatment (§5.6) still applies.

### D12 — [Location + Site] Restore the gear section (Issue 9)
Source: user ("where did the gear section go? the whole point is to let users know the basic set, and if it's a special dive site e.g. terrible current, recommend dive hooks").
Decision: Restore gear on the site detail page using the intact data layer (F7: `siteSpecificGear`, `gearIds`). Model = basic kit (Layer A, adapts to site water temp + skill level) + site-specific add-ons (Layer B, e.g. reef hook for strong current). Matches existing EXPERIENCE.md §7.3 step 12 and journey gear beats (§10.2, §10.4). Place in the centre column after "How to dive this site."
Scope (user confirmed): BOTH surfaces. Site pages carry the full Layer A + Layer B gear list. Location pages show a rolled-up gear SUMMARY — typical basic kit for the region/water temp plus any standout site-specific add-ons aggregated across the location's sites (e.g. "reef hooks needed at 2 sites"). Location summary links down to the relevant site for detail; never invents gear not present in site data.

### D13 — [Site] Clarify centre-vs-sidebar flow (Issue 10)
Source: user ("not exactly duplication, but the flow is confusing — on the right side it's like depth profile, then how to get there").
Decision: Re-sequence the site sidebar into a single logical narrative. Move marine CONDITIONS (thermal/live panel) OUT of the sidebar into the centre "Conditions" section where conditions content already lives. Sidebar becomes a clean planning spine: Depth profile (visual) → Part of [location] (context) → Plan/operators (booking). Result: centre column = what you'll see + conditions + how to dive + gear + about; sidebar = orient + plan. No content rendered in two competing places.
