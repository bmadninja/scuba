# DECISIONS.md — Autonomous Redesign Build
All forks, taste calls, and deviations logged here. Append; never overwrite.

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Kept legacy `--atlas-*` CSS variables as aliases to new light tokens | Existing components use `var(--atlas-bg)`, `var(--atlas-ink)` etc. Remapping them to the new light values avoids a mass search-and-replace across all components while maintaining correct rendered output under the new token system. |
| 2 | `ReefHealthBadge` uses `improving/stable/declining` state names; `ReefStateBadge` keeps `thriving/pressure/change` | The data layer uses `thriving/pressure/change`. The new badge is for display purposes and uses the design spec names. The old badge re-exports the new one alongside itself so both work without touching data files. |
| 3 | Logo updated to text-only wordmark (removed SVG water-drop icon) | Design system rule: no icons in UI (UX-DR20). Logo is now Source Serif 4 "Scuba Season" text. `dark` prop kept for API compatibility. |
| 4 | AtlasNav `HeroNav` variant removed | New nav handles transparent/solid state via scroll detection on a single component. The hero variant was only used on pages that are now on the Epic 2 backlog. Removing it reduces duplicate code and avoids the old `hideLayoutNav` complexity for the nav. |
| 5 | Footer hover colors implemented via `<style>` block CSS (not inline `onMouseEnter`) | AtlasFooter is a Server Component. Event handlers cannot be passed in RSC. CSS `:hover` achieves the same hover-to-yellow effect without requiring `"use client"`. |
| 6 | Fixed pre-existing bug in `src/lib/atlas-location.ts` line 177 | 358/380 sites have no `species` field. `s.species.map(...)` crashed with "Cannot read properties of undefined". Added `?? []` null-coalesce guard. Not a data write — guard is in application code only. |
| 7 | Nav search input uses `rounded-sm` (2px radius) not pill shape | Pill search was part of old Kimi rebrand aesthetic. New design uses rectangular elements (border-radius 2px per spec). |
| 8 | `/explore` route not yet created (Epic 3); nav "Explore" link points to `/locations` | Spec says `/explore` but that route is Epic 3 scope. Using `/locations` as fallback so nav is functional on Epic 1 delivery. |
| 9 | Hero photo uses OceanImageBank_FilippoBorghi_04.jpg (Raja Ampat manta ray) | No local underwater photo asset exists in `/public`. This is the most dramatic underwater shot already referenced in the existing homepage code, confirmed underwater via OceanImageBank source. DECISIONS.md notes this per Epic 2 Story 2.1 requirement. |
| 10 | ReefStateCard trio uses computed reef states (improving/stable/declining) not the data layer's thriving/pressure/change | The data layer returns `thriving/pressure/change` but the design spec and ReefHealthBadge use `improving/stable/declining`. Cards are hardcoded server-side with the mapped display state matching the underlying data computation result. No data files were modified. |
| 11 | HomepageStatStrip is a new Client Component (`homepage-stat-strip.tsx`) separate from existing `stat-strip.tsx` | The existing StatStrip has a different API and styling. Epic 2 spec requires yellow border-y, counter animation, and a different layout. Creating a new component avoids breaking existing stat-strip usages in location/site pages. |
| 12 | SpeciesFilmstrip uses 10 hardcoded species with iNaturalist photo thumbnails | No `src/data/species.json` exists. The species data lives inside each site's `species` array within `sites.json`. Building per-species sighting odds cross-site would require a data aggregation step that is out of scope (data read-only rule). Hardcoded sample data fulfills Story 2.5 display requirements with real iNat photo URLs. |
| 13 | `/submit` route is the Upload CTA destination | The sighting submission feature exists in `src/app/api/` and `src/components/sighting-submission/` (untracked). CTAs point to `/submit` as the presumed submission route. If the route slug differs, CTAs need updating when Epic 5 lands. |
| 14 | Hero bottom gradient fades to white (#FFFFFF not transparent) | The spec says "from-transparent to-white" for the bottom gradient. This creates a clean transition into the yellow-bordered StatStrip. The gradient goes to exact `--color-paper: #FFFFFF`. |
| 15 | Epic 3: new `ExplorePage` component replaces `LocationsExplorer` entirely | The existing `LocationsExplorer` was dark-themed (#0a1628 bg), used the old cyan (#00d4ff) accent, and had no map+grid split layout. Rather than patch it in-place (which would require extensive override), a new `explore-page.tsx` was created implementing all 5 stories with the new light tokens. The old `LocationsExplorer` is kept because it may be referenced elsewhere; the page now imports `ExplorePage`. |
| 16 | Epic 3: `FilterBar` simplified to 3 filter categories (reef state, broad region, month) | The spec asks for reef state + region + best season month pills. The existing explorer had 6+ filter types (cert, "what to see", fresh eyes, etc.). The simplified `FilterBar` is the primary pill bar; the mobile sheet adds the same 3 categories. The removed filter types (cert, wildlife, fresh eyes) are accessible on the old component if needed but out of scope for Epic 3 stories 3.1–3.5. |
| 17 | Epic 3: `hasSightings` computed by checking `siteIds` from base locations.json → `getSightingsBySiteId` | The sightings data is indexed by `siteId`, not locationId. Base `locations.json` has a `siteIds` array. The page computes `hasSightings` server-side by checking if any site of the location has at least 1 sighting record. No data files modified. |
| 18 | Epic 3: `LocationsGlobe` updated to accept optional `onMarkerClick` prop | Previously, all globe marker clicks navigated to `/locations/[slug]`. The explore page needs marker clicks to select and scroll to the corresponding card (not navigate away). Added optional `onMarkerClick?: (slug: string) => void` — when provided it overrides the default navigate behavior. Globe height also made configurable. Backwards-compatible change. |
| 19 | Epic 3: Hover border on `ReefCard` uses global `.reef-card:hover` CSS from globals.css | The existing globals.css already defines `.reef-card:hover { border-color: var(--color-ink) }`. The `ReefCard` component adds `className="reef-card"` to pick up this rule instead of injecting per-card `<style>` tags, which would cause a flash of unstyled border on hydration. |
